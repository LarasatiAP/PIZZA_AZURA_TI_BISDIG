3.4 Perancangan Penerapan Solusi
FLOWCHART
Flowchart Sistem Pizza Azura menggambarkan alur proses pemesanan pizza dari pelanggan, pengelolaan status pesanan oleh sistem, hingga penanganan pesanan oleh admin. Diagram dibagi menjadi tiga swimlane utama agar tanggung jawab tiap pihak terlihat jelas, yaitu Pelanggan (Customer), Sistem/Status Pesanan, dan Admin.
1. Struktur Umum Flowchart
Pelanggan (Customer): berisi aktivitas pengguna saat memesan pizza melalui aplikasi.
Sistem / Status Pesanan: menunjukkan bagaimana sistem menerima dan memperbarui status pesanan secara otomatis.
Admin: menggambarkan proses pengelolaan pesanan oleh pihak restoran.
Alur utama dimulai dari pelanggan membuka aplikasi, memilih menu, melakukan pemesanan, lalu pesanan diproses oleh admin hingga selesai dan diambil oleh pelanggan.
2. Penjelasan Simbol Flowchart
Simbol
Fungsi

Menandakan awal dan akhir proses (Start dan End).
Persegi panjang
Menunjukkan aktivitas atau proses.
Belah ketupat
Menunjukkan keputusan (decision) dengan cabang Yes/No.
Panah
Menunjukkan arah aliran proses.
3. Alur Detail Sistem
A. Proses Pelanggan (Customer)
Start – Buka AplikasiPelanggan membuka aplikasi Pizza Azura untuk memulai pemesanan.
Lihat Daftar MenuSistem menampilkan berbagai pilihan pizza dan menu lainnya yang tersedia.
Pilih Pizza dan Ukuran (S/M/L)Pelanggan memilih jenis pizza serta ukuran yang diinginkan:
S = Small
M = Medium
L = Large
Pilih Topping TambahanPelanggan dapat menambahkan topping ekstra sesuai selera, misalnya keju, sosis, jamur, atau pepperoni.
Tambah ke KeranjangPizza yang dipilih dimasukkan ke dalam keranjang belanja.
Keputusan: “Pesan Lagi?”Sistem menanyakan apakah pelanggan ingin menambah pesanan lain.
Yes: pelanggan kembali ke langkah “Lihat Daftar Menu” untuk menambah item lain.
No: proses lanjut ke tahap checkout.
Buka Keranjang & ReviewPelanggan memeriksa kembali seluruh pesanan, jumlah item, ukuran, topping, dan total harga.
Isi Data PelangganPelanggan mengisi data penting, yaitu:
Nama
Nomor HP
Tipe pesanan (delivery atau take away)
Metode pembayaran
Konfirmasi PesananSetelah data benar, pelanggan mengonfirmasi pesanan. Pada tahap ini pesanan dikirim ke sistem dan masuk ke antrean admin.
End – Pelanggan Ambil PesananProses dari sisi pelanggan berakhir ketika pesanan selesai dan diambil atau diterima.
B. Proses Sistem / Status Pesanan
Pesanan Masuk ke SistemSetelah pelanggan mengonfirmasi, sistem otomatis mencatat pesanan dan membuat antrean baru.
Status: PENDINGPesanan menunggu tindakan admin. Status ini berarti pesanan baru diterima tetapi belum diproses.
Status: DIPROSESSetelah admin menerima pesanan, sistem memperbarui status menjadi diproses. Pesanan mulai disiapkan oleh restoran.
Status: SEDANG DIBUATPesanan sedang dimasak atau dirakit di dapur. Pelanggan dapat melihat progres ini melalui fitur cek status.
Status: SELESAIPesanan telah selesai dibuat dan siap diambil atau dikirim.
Notifikasi SelesaiSistem mengirim notifikasi kepada pelanggan bahwa pesanan sudah siap.
Pelanggan Cek Status (Refresh)Pelanggan dapat memperbarui halaman status untuk melihat perkembangan pesanan secara real-time.
C. Proses Admin
Login AdminAdmin masuk ke sistem menggunakan akun khusus untuk mengelola pesanan.
Lihat Dashboard & AntreanDashboard menampilkan daftar pesanan yang sedang masuk beserta statusnya.
Lihat Pesanan Baru (Pending)Admin memeriksa detail pesanan yang baru diterima.
Keputusan: “Terima Pesanan?”Admin menentukan apakah pesanan dapat diproses.
Yes – Terima Pesanan: admin menerima pesanan dan sistem mengubah status menjadi Diproses.
No – Hapus Pesanan: jika pesanan tidak valid atau tidak dapat dipenuhi, admin menghapus pesanan dari sistem.
Update Status → DiprosesMenandakan pesanan sudah masuk tahap persiapan.
Update Status → Sedang DibuatAdmin memperbarui status ketika pizza sedang dimasak di dapur.
Update Status → SelesaiSetelah pizza siap, admin menandai pesanan sebagai selesai.
Monitor Stok BahanAdmin juga memantau ketersediaan bahan baku seperti keju, saus, dan topping agar operasional tetap lancar.
4. Hubungan Antar Swimlane
Dari Pelanggan ke Sistem: pesanan yang dikonfirmasi dikirim ke sistem dan diberi status Pending.
Dari Sistem ke Admin: pesanan pending muncul di dashboard admin untuk ditindaklanjuti.
Dari Admin ke Sistem: setiap perubahan status oleh admin memperbarui informasi pesanan secara otomatis.
Dari Sistem ke Pelanggan: pelanggan menerima pembaruan status dan notifikasi hingga pesanan selesai.
5. Ringkasan Alur Utama
Pelanggan memilih menu dan melakukan checkout.
Sistem mencatat pesanan dengan status Pending.
Admin menerima pesanan dan mengubah status menjadi Diproses.
Pesanan masuk tahap Sedang Dibuat di dapur.
Setelah selesai, status berubah menjadi Selesai dan pelanggan mendapat notifikasi.
Pelanggan mengambil atau menerima pesanan.
6. Tujuan dan Manfaat Flowchart
Memperjelas alur bisnis pemesanan pizza dari awal hingga akhir.
Membagi tanggung jawab antara pelanggan, sistem, dan admin.
Memudahkan pengembangan aplikasi karena proses sudah terdokumentasi dengan jelas.
Meningkatkan efisiensi operasional melalui pemantauan status pesanan secara real-time.
7. Kesimpulan
Flowchart Sistem Pizza Azura menunjukkan proses pemesanan yang terstruktur dan terintegrasi. Pelanggan dapat memesan pizza dengan mudah, sistem mengelola status pesanan secara otomatis, dan admin menangani pesanan hingga selesai. Dengan adanya pembaruan status dan notifikasi, komunikasi antara restoran dan pelanggan menjadi lebih transparan dan efisien.

Activity Diagram 
Activity Diagram menggambarkan alur kerja (workflow) proses pemesanan pizza dari sisi Customer, proses validasi oleh Sistem, dan proses pengelolaan pesanan oleh Admin.
1. Buka Website Pizza AzuraCustomer mengakses halaman utama melalui browser.
2. Melihat Menu & Filter KategoriCustomer menelusuri daftar pizza dan dapat memfilter berdasarkan kategori (Classic/Standard/Premium).
3. Decision: Pilih Pizza?Jika "Ya" → lanjut ke kustomisasi. Jika "Tidak" → kembali melihat menu.
4. Kustomisasi PizzaCustomer memilih ukuran (S/M/L), menambah topping opsional, mengatur jumlah, dan menulis catatan khusus.
5. Tambah ke KeranjangPizza yang dikustomisasi ditambahkan ke keranjang belanja.
6. Decision: Pesan Lagi?Jika "Ya" → kembali ke menu. Jika "Tidak" → buka keranjang.
7. Review Pesanan di KeranjangCustomer memeriksa daftar item, subtotal, dan total harga. Bisa menghapus item.
8. Isi Form CheckoutMengisi Nama Pelanggan (wajib), No. WhatsApp (opsional), Tipe Pesanan (Dine In/Take Away/Pre-Order), dan Metode Pembayaran (Cash/QRIS/Transfer).
9. Submit PesananMengirim data pesanan ke server.
Swimlane: Sistem
10. Validasi Data PesananSistem memvalidasi kelengkapan data (nama, item pesanan, dll).
11. Decision: Data Valid?Jika "Tidak" → tampilkan error dan kembali ke form. Jika "Ya" → proses pesanan.
12. Generate Order IDMembuat ID unik format AZR-[timestamp base36] (contoh: AZR-TJKLA).
13. Hitung Total & Simpan ke DatabaseMenghitung total harga (harga dasar + topping × quantity) dan menyimpan ke tabel orders, order_items, dan order_item_toppings.
14. Tampilkan KonfirmasiMenampilkan modal sukses dengan Order ID dan nomor antrian.
Swimlane: Admin
15. Terima Pesanan BaruAdmin melihat pesanan baru di dashboard (auto-refresh setiap 5 detik).
16. Update Status PipelineAdmin memperbarui status secara bertahap: Pending → Processing → Cooking → Done.
17. Pesanan SelesaiStatus berubah menjadi "Done", pendapatan tercatat di statistik.
ERD
ERD menggambarkan struktur relasional database Pizza Azura. Terdapat 7 tabel utama yang saling terhubung melalui foreign key constraints.
R1 – orders → order_itemsKardinalitas: 1 : N (One-to-Many)Satu pesanan memiliki satu atau lebih item pizza. FK: order_items.order_id → orders.id dengan ON DELETE CASCADE.
R2 – order_items → order_item_toppingsKardinalitas: 1 : N (One-to-Many)Satu item pesanan bisa memiliki nol atau lebih topping tambahan. FK: order_item_toppings.order_item_id → order_items.id dengan ON DELETE CASCADE.
R3 – menu → order_itemsKardinalitas: 1 : N (Referensi)Item pesanan mereferensikan menu pizza (snapshot nama & gambar). Field: order_items.menu_id → menu.id.
R4 – toppings → order_item_toppingsKardinalitas: 1 : N (Referensi)Topping pesanan mereferensikan master topping. Field: order_item_toppings.topping_id → toppings.id.
adminsKeterangan: StandaloneTabel independen untuk menyimpan kredensial admin.
settingsKeterangan: StandaloneTabel key-value untuk konfigurasi toko.

Proses Checkout
Customer mengisi seluruh data yang diperlukan pada form checkout kemudian menekan tombol "Submit Pesanan".
Sistem melakukan validasi data, termasuk memastikan nama pelanggan telah diisi dan terdapat item di dalam keranjang.
Jika data valid, sistem mengirimkan data ke server melalui endpoint POST /api/orders.
Server menghitung total harga pesanan, membuat Order ID unik dengan format AZR-xxxx, kemudian menyimpan data ke database.
Jika proses berhasil, sistem menampilkan modal Order Success sebagai konfirmasi bahwa pesanan telah berhasil dibuat.
